# Development Guide

## Tech Stack

### Frontend
Because our app is developed to be iOS native, our frontend is written using the SwiftUI framework from the Swift programming language. SwiftUI is incredibly easy to use and produces apps closely aligned with the iOS design language. In addition, the API calls that interface with our backend from the frontend are also written in Swift. The data structures that data is packaged in in the frontend are defined within the `Shared` folder in the `Structs.swift` file as well as the `Models.swift` file, along with some mock data for these structs. The `APIClient.swift` file contains the functions that retrieve data from our backend; data is generally received into an `APIUser` type file, after which its converted into the `User` struct that our frontend handles. To maintain its nativity, further development should remain in Swift, all the code for which can be found in the `TutorNation` folder. Due to Swift's tight coupling with Apple, our build tool for our app's frontend is through the Xcode IDE as well. 

### Backend
We chose Spring Boot, a Java framework for making Spring applications, for our backend. It provides most of the components needed for creating a hostable application, so it made the process of setting up a backend fairly smooth. Most of our Spring Boot code is contained within the `TutorNation_Backend` folder. Much of the code was generated by the framework, but the controllers that reach over to our database and pull information down is in the `src/main/java/com/tutornation/backend` folder. The folder controller has controllers for `Meeting`, `Parent`, `Student`, `Tutor`, and `User`. To deploy our backend, we use the Amazon Elastic Compute Cloud (EC2) for cloud hosting capabilities.

### Infrastructure
For our database, we use Amazon Web Services Relation Database Service (RDS). This is a production level PostgreSQL database configured via `set-env.sh`. For database schema migrations from the localhost to the cloud hosted backend, we use Flyway. The migration files for which can be find in the `src/main/resources/db/migration` folder of the `TutorNation_Backend`.

As for containerization, we choose docker as its an industry standard and makes packaging the starting of our Spring application incredibly straightforward. This is also aided by Apache Maven, our build automation tool. Our database and our backend application are both orchestrated by Docker Compose.

### Building and Running
From top to bottom, to build and run our app you should do the following:
- From the `TutorNation_Backend` folder, run the command `docker compose up --build` to start the local backend and database.
- Open Xcode using the `TutorNation.xcodeproj` file after cloning the repo.
- Select a simulator compatible with iOS target version 18+ and click the Run button in the toolbar or Product -> Run.
There is currently no CI/CD in our project repo to automate testing, building, deploying, etc. upon new pushes.

## Style

### Swift / Frontend
- **Naming:** UI-facing models use `camelCase` properties (`firstName`, `lastName`). API-facing structs use `snake_case` to match the backend JSON (`first_name`, `last_name`).
- **File organization:** Views are organized by role (`Student/`, `Tutor/`, `Parent/`), shared models and API code go in `Shared/`, and reusable UI components go in `Components/`. New views should follow the same structure.
- **View structure:** Large views extract sections into private computed properties (e.g., `LoginPage.swift` has `header`, `modePicker`, `formCard`, `socialButtons`). This pattern allows for much more digestible frontend views as they grow in size and complexity. 
- **API layer:** All backend calls go through `APIClient.shared`. New endpoints should be added there following the same pattern of generic `getRequest`/`postRequest`/`putRequest` helpers.
- **Model conversion:** API models have `.toStudent()`, `.toUser()`, etc. extension methods that convert to UI-facing models. Do not use API structs directly in views.

### Java / Backend
- **Controllers use raw JDBC** (`JdbcTemplate` with SQL strings and `Map<String, Object>`) rather than JPA entities or typed DTOs. This is the established pattern — new endpoints should stay consistent with it.
- **Routing convention:** Controllers are mapped under `/api/{resource}` (e.g., `/api/students`, `/api/tutors`). Each entity gets its own controller class.
- **Package structure:** All controllers live in `com.tutornation.backend.controller`. If services or repositories are added later, they should follow standard Spring package conventions (`service/`, `repository/`).

### Database
- **Flyway migrations** use the naming convention `V{N}__{description}.sql` in `src/main/resources/db/migration/`. New migrations should increment from the current highest version number. Do not modify previous files as that will mess up the Flyway migrations
- **Column naming** uses `snake_case` throughout (e.g., `first_name`, `created_at`, `password_hash`).

## Testing
There are two relevant files to testing this program. First, our plan for testing can be find in the `TESTPLAN.md` file detailing everything from the purpose and scope of our testing, to the pass/fail guidelines and test deliverables.

Along with the `TESTPLAN.md` file, we also include the `TESTCASES.md` file. Provided in this file is a suite of test cases ensuring the proper functioning of the API, authentication, role selection, onboarding, and just about every other feature of the app related to interfacing with the backend and database. To test this app, follow the cases outlined in the `TESTCASES.md` file ensuring the correct preconditions are met and the steps involved in the test are properly followed.
